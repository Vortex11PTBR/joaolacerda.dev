Fundamentos que parecem simples mas caem
GROUP BY, HAVING, subqueries, NULL
0/5
▼
Diferença entre WHERE e HAVING
fácil
pergunta real"Qual a diferença entre WHERE e HAVING? Quando usar cada um?"
Resposta: WHERE filtra linhas ANTES do agrupamento. HAVING filtra grupos DEPOIS do GROUP BY. WHERE não pode usar funções de agregação (COUNT, SUM) — HAVING pode. Exemplo: SELECT dept, COUNT(*) FROM employees GROUP BY dept HAVING COUNT(*) > 5
Como NULL se comporta em comparações
médio
pegadinha clássica"O que retorna: SELECT * FROM tabela WHERE coluna != 'valor'? Retorna registros com NULL?"
Resposta: Não. NULL não é igual nem diferente de nada — qualquer comparação com NULL retorna NULL (nem true nem false). Para filtrar NULL use IS NULL ou IS NOT NULL. Isso pega muita gente desprevenida em produção.
Subqueries correlacionadas
médio
pergunta real"Escreva uma query que retorne os funcionários que ganham mais que a média do seu departamento."
Solução clássica com subquery correlacionada: SELECT e1.nome, e1.salario, e1.dept
FROM funcionarios e1
WHERE e1.salario > (
  SELECT AVG(e2.salario)
  FROM funcionarios e2
  WHERE e2.dept = e1.dept
)
DISTINCT vs GROUP BY — diferença real
fácil
DISTINCT remove linhas duplicadas no resultado. GROUP BY agrupa linhas para aplicar funções de agregação. Funcionalmente para contar únicos são equivalentes, mas GROUP BY é mais flexível (você pode adicionar COUNT, SUM, etc). Performance pode variar por banco.
ORDER de execução de uma query SQL
difícil
pergunta de sênior que cai em junior"Em que ordem o banco executa as cláusulas de um SELECT?"
Ordem de execução: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT. Por isso você não pode usar alias do SELECT no WHERE — o WHERE roda antes do SELECT ser processado.
JOINs — o favorito dos entrevistadores
INNER, LEFT, RIGHT, FULL, SELF, CROSS
0/4
▼
INNER vs LEFT JOIN — quando usar cada um
fácil
pergunta eliminatória"Qual a diferença entre INNER JOIN e LEFT JOIN? Me dê um exemplo de quando usar cada."
INNER JOIN: só retorna linhas que têm correspondência nas duas tabelas. Use quando você só quer dados que existem nos dois lados.

LEFT JOIN: retorna TODAS as linhas da tabela esquerda, com NULL onde não há correspondência à direita. Use quando você quer todos os registros mesmo sem match — ex: todos os clientes, mesmo os que nunca compraram.
SELF JOIN — quando e como usar
médio
pergunta real"Como você buscaria o nome do gerente de cada funcionário, sendo que ambos estão na mesma tabela?"
SELECT e.nome, g.nome AS gerente
FROM funcionarios e
LEFT JOIN funcionarios g ON e.gerente_id = g.id
Encontrar registros sem correspondência (anti-join)
médio
clássico"Liste todos os clientes que nunca fizeram um pedido."
SELECT c.nome
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
WHERE p.id IS NULL
O truque é o LEFT JOIN + WHERE NULL na chave do lado direito.
UNION vs UNION ALL
fácil
UNION remove duplicatas (mais lento — faz sort). UNION ALL mantém tudo (mais rápido). Use UNION ALL por padrão e só use UNION quando precisar de unicidade — a diferença de performance em tabelas grandes é significativa.
Window Functions — diferencial de junior
ROW_NUMBER, RANK, LAG, LEAD, PARTITION BY
0/3
▼
ROW_NUMBER vs RANK vs DENSE_RANK
médio
diferença críticaDado empate no valor: ROW_NUMBER sempre dá números únicos (1,2,3,4). RANK pula números após empate (1,1,3,4). DENSE_RANK não pula (1,1,2,3). Na prática: use DENSE_RANK para rankings de negócio (top 3 vendedores).
LAG e LEAD — comparar com linha anterior/próxima
médio
pergunta real de dados"Como calcular a variação de receita mês a mês em SQL?"
SELECT mes, receita,
  LAG(receita) OVER (ORDER BY mes) AS mes_anterior,
  receita - LAG(receita) OVER (ORDER BY mes) AS variacao
FROM vendas
Top N por grupo com PARTITION BY
difícil
clássico de entrevista"Retorne os 2 funcionários mais bem pagos de cada departamento."
SELECT * FROM (
  SELECT nome, dept, salario,
    ROW_NUMBER() OVER (
      PARTITION BY dept ORDER BY salario DESC
    ) AS rn
  FROM funcionarios
) t WHERE rn <= 2
CTEs, Performance e Índices
WITH, EXPLAIN, índices compostos
0/3
▼
CTEs (WITH) — legibilidade e reuso
fácil
CTE é um resultado temporário nomeado dentro da query. Torna queries complexas legíveis. Pode referenciar a mesma CTE múltiplas vezes. CTEs recursivas são usadas para hierarquias (org chart, categorias aninhadas). Prefira CTE a subquery aninhada para manutenção.
O que é um índice e quando ele NÃO ajuda
médio
pergunta real"Você adicionaria um índice em toda coluna usada em WHERE? Por quê não?"
Índice acelera leitura mas penaliza escrita (INSERT/UPDATE/DELETE mantém o índice). Em tabelas com muita escrita e pouca leitura, índice pode ser contraproducente. Índice também não ajuda quando a seletividade é baixa (ex: coluna booleana — metade da tabela vai retornar de qualquer forma).
EXPLAIN ANALYZE — ler um plano de execução
difícil
Saiba o que significa: Seq Scan (tabela inteira — ruim em tabelas grandes), Index Scan (usando índice — bom), Nested Loop / Hash Join / Merge Join (tipos de join). O número de rows estimated vs actual rows mostra se as estatísticas do banco estão atualizadas. Só mencionar que você usa EXPLAIN já impressiona entrevistador.

Python Core que cai em dados
List comprehension, generators, tipos mutáveis
0/4
▼
List comprehension vs loop for vs map
fácil
pergunta real"Qual a diferença entre list comprehension, map() e um for loop? Quando usar cada?"
List comprehension é mais pythonico e geralmente mais rápido que for loop. map() é lazy (retorna iterator) — melhor para datasets grandes. Generator expression (parênteses no lugar de colchetes) é ainda mais eficiente em memória. Regra: se vai usar o resultado inteiro, list comp. Se vai iterar uma vez ou o dado é grande, generator.
Mutável vs Imutável — por que importa em dados
médio
pegadinha"O que acontece quando você passa uma lista como argumento padrão de uma função?"
def adicionar(item, lista=[]): # BUG!
  lista.append(item)
  return lista
A lista padrão é criada UMA vez e reutilizada entre chamadas. Bug clássico. Solução: usar None como padrão e criar a lista dentro da função.
Context managers — with open() e por que usar
fácil
O bloco with garante que o recurso é fechado mesmo se ocorrer exceção. Fundamental para arquivos, conexões de banco, locks. Em pipelines de dados, não fechar conexões corretamente causa memory leak em produção.
Decorators — o que são e onde aparecem em dados
médio
Decorator é uma função que envolve outra função para adicionar comportamento. Em dados você vai ver decorators em frameworks como Prefect (@flow, @task), FastAPI (@app.get) e pytest (@pytest.fixture). Entenda o conceito, não precisa criar do zero.
Pandas — o essencial
O que entrevistadores de dados junior realmente testam

loc vs iloc — diferença real
fácil
loc seleciona por label/nome do índice e colunas. iloc seleciona por posição inteira (0, 1, 2...). Confundir os dois é bug clássico quando o índice não é sequencial (ex: após filtrar um DataFrame, os índices não reiniciam automaticamente).
groupby() + agg() — agregações complexas
médio
teste prático comum"Dado um DataFrame de vendas, calcule por produto: total de vendas, média de preço e número de transações."
df.groupby('produto').agg(
  total=('valor', 'sum'),
  media=('preco', 'mean'),
  transacoes=('id', 'count')
).reset_index()
Lidar com dados faltantes — estratégias
médio
pergunta conceitual"Como você lida com valores nulos em um dataset? Sempre dropa ou sempre imputa?"
Depende do contexto. Dropar: quando a % de nulos é pequena e aleatória. Imputar com média/mediana: variáveis numéricas sem outliers extremos. Imputar com moda: variáveis categóricas. Imputar com modelo: quando os nulos têm padrão. Manter como categoria: quando "ausente" é informação em si (ex: cliente que não preencheu renda).
merge() — equivalente ao SQL JOIN
fácil
pd.merge(df1, df2, on='id', how='left') é o LEFT JOIN do Pandas. Os how possíveis: 'inner', 'left', 'right', 'outer'. Saiba a diferença entre merge (horizontal, por chave) e concat (vertical, empilhar DataFrames).

Medidas de tendência central e dispersão
Média, mediana, moda, desvio padrão, variância
0/4
▼
Quando a média mente — a pergunta favorita
fácil
pergunta real"Salário médio do Brasil é R$2.800. Isso quer dizer que metade das pessoas ganha mais que isso?"
Não. Distribuição de renda é assimétrica à direita — poucos milionários puxam a média para cima. A mediana é mais representativa em distribuições assimétricas. Exemplo: 9 pessoas ganham R$ 1.000 e 1 ganha R$ 100.000. Média = R$ 10.900. Mediana = R$ 1.000. Saber isso em entrevista de dados faz diferença.
Desvio padrão vs variância — o que realmente significa
fácil
Variância é a média dos quadrados das diferenças da média — unidade é quadrada (difícil de interpretar). Desvio padrão é a raiz da variância — mesma unidade dos dados, fácil de interpretar. Regra prática: em distribuição normal, 68% dos dados estão a 1 desvio padrão da média, 95% a 2 desvios. Use isso para identificar outliers.
Correlação não é causalidade — exemplos práticos
médio
pergunta conceitual"Encontramos correlação de 0.85 entre consumo de sorvete e afogamentos. Devemos reduzir venda de sorvete?"
Variável confundidora: temperatura. Calor causa tanto consumo de sorvete quanto idas à praia (afogamentos). Correlação mede relação linear entre variáveis — não implica que uma causa a outra. Sempre pergunte: existe uma terceira variável explicando ambas?
Outliers — identificar e decidir o que fazer
médio
Método IQR: outlier é valor abaixo de Q1 - 1.5*IQR ou acima de Q3 + 1.5*IQR. Mas antes de remover, pergunte: é erro de coleta ou é informação real? Uma venda de R$ 500.000 num dataset de varejo pode ser um cliente corporativo legítimo — remover seria perder informação de negócio.

Conceitos fundamentais de pipeline
ETL, ELT, batch, streaming, idempotência
0/5
▼
ETL vs ELT — por que o mundo migrou para ELT
fácil
ETL: Extract → Transform → Load. Transforma antes de carregar. Era o padrão quando storage era caro. ELT: Extract → Load → Transform. Carrega cru primeiro, transforma depois no warehouse. Possível com warehouses modernos baratos (BigQuery, Snowflake). ELT é mais flexível — você pode retransformar sem re-extrair os dados.
Idempotência — por que todo pipeline precisa disso
médio
pergunta que separa junior de pleno"O que é um pipeline idempotente e por que isso importa?"
Uma operação é idempotente quando pode ser executada múltiplas vezes com o mesmo resultado. Pipelines falham — o orquestrador vai re-executar. Se seu pipeline não é idempotente, você terá dados duplicados. Solução: usar INSERT ... ON CONFLICT DO UPDATE (upsert) em vez de INSERT simples.
Batch vs Streaming — quando usar cada
fácil
Batch: processa dados em lotes periódicos (hourly, daily). Simples, barato, suficiente para relatórios. Streaming: processa evento a evento em tempo real (Kafka, Flink). Necessário para detecção de fraude, alertas em tempo real. A maioria das empresas começa com batch e migra para streaming quando a latência importa.
O que é dbt e por que virou padrão
fácil
dbt (data build tool) transforma SQL em código versionável com testes, documentação e dependências entre modelos. É o "Git para transformações SQL". Você escreve SQL SELECT, o dbt cria as tabelas/views. Habilidade muito pedida em vagas de analytics engineer e data engineer em fintechs.
Data Lake vs Data Warehouse vs Data Lakehouse
médio
Data Lake: armazena dados brutos em qualquer formato (S3, GCS). Barato, flexível, mas vira "data swamp" sem governança. Data Warehouse: dados estruturados, schema definido, otimizado para query (BigQuery, Snowflake, Redshift). Caro mas rápido. Lakehouse: tenta combinar os dois (Delta Lake, Apache Iceberg) — storage de lake com capacidade analítica de warehouse.

Event Loop & Assincronia
O conceito mais cobrado em entrevistas JS
0/4
▼
O que é o Event Loop — explicar sem travar
crítico
a pergunta"Me explica como o Event Loop funciona no JavaScript."
JS é single-threaded — só executa uma coisa por vez. A Call Stack executa código síncrono. Operações assíncronas (fetch, setTimeout) vão para Web APIs. Quando terminam, o callback vai para a Task Queue (ou Microtask Queue para Promises). O Event Loop fica verificando: se a Call Stack está vazia, pega o próximo item da fila. Microtasks (Promises) têm prioridade sobre Macrotasks (setTimeout).
Promise vs async/await — o que acontece por baixo
médio
async/await é açúcar sintático para Promises. Uma função async sempre retorna uma Promise. await pausa a execução da função sem bloquear a thread. Erro clássico: usar await dentro de forEach (não funciona como esperado — use for...of ou Promise.all). Saber a diferença entre Promise.all, Promise.allSettled e Promise.race impressiona.
O que é hoisting
fácil
Hoisting é o comportamento do JS de "elevar" declarações para o topo do escopo antes da execução. var é hoisted e inicializado como undefined. let e const são hoisted mas ficam em Temporal Dead Zone até a declaração. Funções declaradas com function são hoisted inteiras. Arrow functions não.
Closure — o que é e onde aparece
médio
pergunta clássica"O que é closure? Me dê um exemplo prático."
Closure é quando uma função "lembra" do escopo onde foi criada, mesmo após esse escopo ter encerrado. function contador() {
  let count = 0;
  return () => ++count; // closure sobre count
}
const inc = contador();
inc(); // 1, inc(); // 2Aparece em React (hooks são closures), debounce/throttle, módulos.
this, prototype e ES6+
Conceitos que confundem junior
0/3
▼
this — por que é confuso e como dominar
difícil
this depende de COMO a função é chamada, não onde foi definida. Regra: método de objeto → this = objeto. Função normal → this = global (ou undefined em strict mode). Arrow function → this = escopo léxico (herda de fora). Classe → this = instância. Por isso handlers de evento em React precisam de bind ou arrow function.
Spread, rest, destructuring — o básico que sempre cai
fácil
Spread: [...arr] copia array, {...obj} copia objeto (shallow). Rest: function f(...args) agrupa argumentos em array. Destructuring: const {name, age} = user. Esses três aparecem em todo código React — dominar é obrigatório.
== vs === — nunca use ==
fácil
== faz coerção de tipo (0 == false é true, '' == false é true). === compara tipo E valor. Use sempre ===. Isso é tão básico que se errar, o entrevistador vai desconfiar do resto do seu conhecimento.

React fundamentals
Hooks, renderização, reconciliation
0/5
▼
useState — quando re-renderiza e quando não
médio
pegadinha"Se eu fizer setState com o mesmo valor atual, o componente re-renderiza?"
React usa Object.is para comparação. Se o novo valor é idêntico (primitivos) ou a mesma referência (objetos/arrays), React pula a re-renderização. Por isso setItems([...items, newItem]) e não items.push(newItem); setItems(items) — o push muta o mesmo array (mesma referência), React acha que não mudou.
useEffect — dependências e cleanup
médio
useEffect roda após cada render por padrão. Array de dependências controla quando roda. Array vazio = só na montagem. Sem array = todo render. Cleanup: a função retornada pelo useEffect roda antes do próximo efeito e na desmontagem. Essencial para cancelar fetches, limpar timers e remover event listeners — sem isso, memory leak.
useMemo e useCallback — quando usar e quando é overkill
médio
useMemo: memoiza o resultado de um cálculo. useCallback: memoiza a referência de uma função. Só use quando: (1) o cálculo é realmente caro, ou (2) você está passando a função como prop para um componente memoizado com React.memo. Usar em tudo é otimização prematura — tem custo de memória.
Por que a key em listas importa tanto
fácil
pergunta real"Por que usar index como key em uma lista é um problema?"
React usa keys para identificar qual item mudou durante a reconciliation. Se você usa index como key e reordena ou remove itens, React associa os estados/refs errados aos componentes. Exemplo: lista de inputs — reordenar com index como key vai misturar os valores dos campos. Use sempre um ID único e estável.
SSR vs CSR vs SSG — o que Next.js resolve
médio
CSR: HTML vazio, JS carrega e renderiza no cliente. Ruim para SEO, primeiro load lento. SSR: HTML renderizado no servidor a cada request. Bom para SEO e dados dinâmicos. SSG: HTML gerado em build time. Ultra rápido, mas dados estáticos. Next.js deixa você misturar por página — homepage SSG, dashboard CSR, página de produto SSR.

HTTP & REST
O que todo fullstack precisa saber
0/5
▼
Status codes — os que você precisa saber de cor
fácil
2xx sucesso: 200 OK, 201 Created, 204 No Content. 3xx redirect: 301 Moved Permanently, 302 Found. 4xx client error: 400 Bad Request, 401 Unauthorized (não autenticado), 403 Forbidden (autenticado mas sem permissão), 404 Not Found, 422 Unprocessable Entity (validação), 429 Too Many Requests. 5xx server error: 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable. 401 vs 403 é pegadinha clássica.
PUT vs PATCH — idempotência na prática
médio
pergunta real"Qual a diferença entre PUT e PATCH? Quando usar cada?"
PUT: substituição total do recurso. Você envia o objeto inteiro. PATCH: atualização parcial. Você envia só os campos que mudaram. PUT é idempotente (chamar duas vezes tem o mesmo resultado). PATCH pode ou não ser idempotente dependendo da implementação.
JWT — como funciona de verdade
médio
pergunta real"JWT é seguro? O que está dentro do token? Como o servidor valida?"
JWT tem 3 partes: Header (algoritmo), Payload (claims — userId, role, exp) e Signature. O payload é apenas Base64 — qualquer um pode decodificar. A segurança está na assinatura, que só quem tem a secret key pode gerar. O servidor valida a assinatura — não precisa de banco. Nunca colocar dados sensíveis no payload.
CORS — o erro que todo junior já viu
fácil
CORS (Cross-Origin Resource Sharing) é uma política de segurança do browser. Quando frontend em localhost:3000 chama API em localhost:4000, o browser bloqueia por origens diferentes. Solução: o servidor configura headers Access-Control-Allow-Origin. É proteção do browser, não do servidor — CORS não protege chamadas server-to-server.
REST vs GraphQL — quando cada um faz sentido
médio
REST: simples, cacheável, cada recurso tem endpoint próprio. Problema: over-fetching (receber mais dados que precisa) e under-fetching (precisar de múltiplas chamadas). GraphQL: cliente define exatamente o que quer. Flexível para UIs complexas. Mais complexo de implementar e cachear. Use REST para APIs públicas e simples. GraphQL para produtos com múltiplos clients (web, mobile) com necessidades diferentes.

O problema N+1 — o bug de performance mais comum
médio
pergunta real"O que é o problema N+1 em ORMs? Como resolver?"
Você busca N posts (1 query) e para cada post busca o autor (N queries). Total: N+1 queries. Resolve com eager loading / JOIN: include: { author: true } no Prisma ou eager_load no ActiveRecord. Derruba aplicações em produção quando a tabela cresce — todo dev fullstack precisa saber isso.
Transações ACID — o que cada letra significa
médio
Atomicity: tudo ou nada. Consistency: dados sempre em estado válido. Isolation: transações concorrentes não se interferem. Durability: dados committed sobrevivem a falhas. Exemplo prático: transferência bancária — debita A e credita B numa transação. Se o sistema cair no meio, o rollback garante que não some dinheiro.
SQL vs NoSQL — quando cada um faz sentido
fácil
Use SQL quando: dados têm relações claras, consistência é crítica (financeiro), queries complexas são frequentes. Use NoSQL quando: schema muda muito, escala horizontal extrema é necessária, dados são documentos sem relações fortes (logs, sessões, catálogos). A maioria das startups começa com PostgreSQL e nunca precisa mudar.
Migrations — por que são inegociáveis
fácil
Migration é um arquivo versionado que descreve uma mudança no schema do banco. Sem migrations: você altera o banco na mão, o colega tem o schema diferente, produção fica diferente de staging. Com migrations (Prisma Migrate, Flyway): todos têm o mesmo schema, mudanças são rastreáveis, rollback possível. Mencionar isso em entrevista mostra maturidade.

Container vs VM — explicar a diferença
fácil
pergunta real"Qual a diferença entre um container Docker e uma VM?"
VM virtualiza hardware completo — tem SO próprio, kernel próprio, mais pesada. Container compartilha o kernel do host — mais leve, inicia em segundos. Docker é como um processo isolado com o ambiente dele, não uma máquina inteira. Por isso container Node.js começa em 1s, VM em 30s.
Dockerfile — o que cada instrução faz
fácil
FROM: imagem base. WORKDIR: diretório de trabalho. COPY: copia arquivos. RUN: executa comando durante o build (instala dependências). ENV: variável de ambiente. EXPOSE: documenta a porta (não publica). CMD: comando padrão ao iniciar o container. Importante: cada instrução cria uma layer — colocar COPY package.json antes de COPY . usa o cache de npm install quando só o código muda.
Variáveis de ambiente — por que nunca hardcodar
fácil
Credenciais no código = vazamento quando você commita. Use .env para desenvolvimento local (no .gitignore), variáveis de ambiente no servidor de produção, e secrets gerenciados (GitHub Secrets, Doppler) no CI/CD. A diferença entre dev/staging/prod está quase sempre nas variáveis de ambiente, não no código.
