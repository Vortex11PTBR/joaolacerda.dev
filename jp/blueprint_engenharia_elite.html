Este blueprint é um ritual de engenharia — abra antes de qualquer projeto, independente do tamanho. Cada fase representa o padrão de excelência praticado em FAANG, fintechs Tier-1 e scale-ups europeus.

Itens marcados como crítico (3 pontos dourados) são inegociáveis. Os demais escalam com a complexidade do projeto. Um projeto alinhado a este padrão fala por si mesmo — sem precisar de palavras.

Estratégia & Produto
Definição e Concepção
0/0
▼
Problem Framing
Problem Statement documentado
Uma frase clara: "Usuário X não consegue Y porque Z, o que causa impacto W". Sem isso, o projeto não começa.
crítico
Design Doc / RFC escrito
Documento textual com: contexto, proposta de solução, alternativas consideradas e trade-offs. Template no estilo Google/Stripe.
críticodocs
Personas e Stakeholders mapeados
Quem usa (User Persona), quem paga (Buyer Persona), quem aprova (Stakeholder). Restrições legais, de negócio e técnicas por persona.
produto
Métricas de sucesso definidas (KPIs / OKRs)
Como saber que o projeto foi bem-sucedido? Ex: latência < 200ms, NPS > 50, churn < 3%. Mensuráveis e com prazo.
críticoproduto
Escopo: MVP definido com limites claros
Lista explícita de "in scope / out of scope" para esta versão. Evita scope creep que mata projetos.
produto
Constraints & Compliance
Restrições LGPD/GDPR mapeadas no início
Que dados pessoais são tratados? Base legal? Titular tem direito ao esquecimento? Isso define a arquitetura de dados.
críticocompliance
Constraints de performance definidos como requisitos
Performance não é opcional. Ex: P99 de latência, throughput alvo, tamanho máximo de payload, budget de custo de infra.

Arquitetura & Design
Planejamento de Sistema
0/0
▼
Modelagem
Diagrama C4 Model (nível 1 e 2 mínimo)
Contexto (quem usa o sistema e o que ele toca) + Containers (aplicações, bancos, serviços). Nível 3 para componentes complexos.
críticoarch
ERD / Schema de dados documentado
Entidades, relacionamentos, cardinalidade, campos obrigatórios/opcionais. Decisão SQL vs NoSQL com justificativa registrada.
críticodados
Domain-Driven Design: Bounded Contexts definidos
Quais são os domínios de negócio? Cada domínio tem sua linguagem ubíqua (Ubiquitous Language). Evita o "big ball of mud".
arch
Estratégia de Database Migration definida
Flyway, Liquibase ou Prisma Migrate. Migrações versionadas no VCS. Zero-downtime migration para produção.
dados
Contratos e Decisões
API Contract First (OpenAPI/Swagger)
Especificação da API definida ANTES do código. Geração de mocks, validação de contratos com Pact. Frontend e Backend desacoplados.
críticoapi
ADRs (Architecture Decision Records) iniciados
Arquivo /docs/adr/. Formato: contexto → decisão → consequências. Por que PostgreSQL e não MongoDB? Por que REST e não gRPC?
críticodocs
SLOs/SLAs definidos antes de codar
Service Level Objectives: uptime alvo (99.9% = ~8h downtime/ano), latência P95/P99, error budget. Engenheiros de Google chamam isso de "reliability contract".
reliability
Threat Model (STRIDE) esboçado
Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege. Mapear ameaças antes de implementar segurança.
security

Implementação & Qualidade
Execução de Código
0/0
▼
Arquitetura de Código
Clean Architecture / Arquitetura em camadas
Domain (regras de negócio), Application (use cases), Infrastructure (DB, APIs externas), Presentation (HTTP, CLI, UI). Dependências sempre apontam para dentro.
críticoarch
Tipagem estrita ativada e sem "any"
TypeScript strict: true. Python com mypy/pyright. Go é tipado por natureza. Banir tipos dinâmicos em código de produção elimina uma classe inteira de bugs.
críticoqualidade
Error Handling explícito e sem try/catch genérico
Erros como valores (Result<T, E>), tipos de erro distintos por domínio. Nunca swallow exceptions. Log contextual em cada erro com trace ID.
críticoqualidade
Validação de input na borda (schema validation)
Zod, Joi, Pydantic, class-validator. Nunca confiar em dados externos sem validar. Validação na entrada, não espalhada no domínio.
qualidade
Developer Experience (DX)
Linter + Formatter configurados (ESLint/Prettier/Ruff)
Padrões de código automáticos. Nenhum PR passa com lint errors. Editor config (.editorconfig) para consistência entre IDEs.
críticodx
Pre-commit hooks com Husky/Lefthook
Lint, format e testes unitários rodando antes de qualquer commit. Commitlint para Conventional Commits. "Se não passa localmente, não vai para o CI".
dx
Conventional Commits + CHANGELOG automatizado
feat:, fix:, chore:, docs:, refactor:, perf:, test:, breaking change!. Ferramentas como semantic-release ou release-please geram CHANGELOG e versionamento automático.
críticogit
PR Template com checklist de revisão
/.github/pull_request_template.md: O que muda? Por quê? Como testar? Há breaking changes? Tem testes? Screenshots (se UI). Faz reviewer pensar.
git
Feature Flags implementados para entrega contínua
Features novas atrás de flags (LaunchDarkly, Unleash, ou simples env vars). Permite deploy sem release, rollback instantâneo e canary releases.
ci/cd
Internacionalização (i18n) preparada desde o início
Mesmo que só PT-BR hoje, estrutura i18n desde o dia 1 evita refactor doloroso. Chaves de tradução em arquivos separados (i18next, Fluent, etc.).
dx

Resiliência & Escalabilidade
Engenharia de Sistemas
0/0
▼
Comunicação e Dados
Comunicação assíncrona para processos pesados
Kafka, RabbitMQ, SQS ou BullMQ. Qualquer operação > 500ms ou que possa falhar vai para fila. Padrões: at-least-once, idempotência, dead letter queue.
críticoarch
Cache Strategy definida com políticas de invalidação
Redis/Memcached. Estratégia: cache-aside, write-through ou write-behind. TTL definido por caso de uso. Cache stampede prevention (mutex key).
perf
Transações ACID e controle de concorrência
Nenhuma operação crítica sem transação. Optimistic locking para alta concorrência. Isolation level adequado (READ COMMITTED vs SERIALIZABLE) com justificativa.
críticodados
Paginação e limites em todas as queries
Cursor-based pagination para datasets grandes. Nenhum SELECT * sem LIMIT em produção. Query timeout definido. EXPLAIN ANALYZE em queries críticas.
perf
Resiliência
Circuit Breaker e Retry com backoff exponencial
Chamadas a serviços externos têm: timeout, retry com jitter, circuit breaker (closed/open/half-open). Biblioteca: Resilience4j, Polly, ou cockatiel.
críticoreliability
Rate Limiting e Throttling na API
Por IP, por usuário, por endpoint. Algoritmos: token bucket ou sliding window. Headers padrão: X-RateLimit-Limit, X-RateLimit-Remaining, Retry-After.
reliability
Graceful Shutdown implementado
SIGTERM → parar de aceitar novas conexões → aguardar requests em voo → fechar conexões de DB → exit(0). Kubernetes mata pods sem perda de dados.
reliability
Health Check endpoints (/health, /ready, /live)
/health: está o processo vivo? /ready: pode receber tráfego? (DB conectado, cache up). Kubernetes usa isso para liveness/readiness probes.
ops

Segurança & Compliance
Estrutura Enterprise
0/0
▼
Autenticação & Autorização
Autenticação via OAuth2/OIDC ou JWT com rotação
Nunca reinventar auth. Tokens de curta duração (access 15min) + refresh token com rotação. Revogação de tokens em logout. Blacklist para tokens comprometidos.
críticosecurity
RBAC ou ABAC implementado
Role-Based ou Attribute-Based Access Control. Verificação de permissão no server, nunca só no client. Princípio do menor privilégio por padrão.
críticosecurity
Audit Log de ações sensíveis
Quem fez o quê, quando e de qual IP. Imutável (append-only). Essencial para LGPD, compliance e investigação de incidentes. Stripe e AWS são referência.
securitycompliance
OWASP & Proteção de Dados
OWASP Top 10 verificado: SQLi, XSS, CSRF, IDOR
Prepared statements (nunca concatenar SQL). CSP headers. CSRF tokens em formulários. IDOR: validar que o recurso pertence ao usuário autenticado.
críticosecurity
Dados sensíveis criptografados em repouso e em trânsito
HTTPS/TLS 1.3 obrigatório. Dados PII criptografados no DB (AES-256). Senhas com bcrypt/argon2 (nunca MD5/SHA1). PCI-DSS se houver pagamentos.
críticosecurity
Secret Management: Zero secrets no código
AWS Secrets Manager, HashiCorp Vault, Doppler ou GitHub Secrets. Nunca em .env commitado. git-secrets ou gitleaks no CI para detectar vazamentos.
críticosecurity
Dependency scanning automatizado (CVEs)
Dependabot, Snyk ou npm audit no CI. Vulnerabilidades críticas bloqueiam o deploy. SBOM (Software Bill of Materials) para projetos enterprise.
security

Qualidade & DevOps
Pipeline de Entrega
0/0
▼
Testes
Test Pyramid: unitários > integração > E2E
70% unitários (rápidos, isolados), 20% integração (DB, APIs), 10% E2E (Playwright/Cypress). Fuja do "ice cream cone" (mais E2E que unitário).
críticoquality
Contract Testing para integrações externas
Pact ou schema validation. Garante que mudanças no provider não quebram o consumer silenciosamente. Crítico em microsserviços.
quality
Coverage mínimo configurado (80% em domínio)
Coverage não é métrica de qualidade, mas é piso de segurança. Foco em coverage do core domain, não de infraestrutura. CI falha abaixo do threshold.
quality
Load Testing antes de cada release major
k6, Artillery ou Gatling. Simular carga esperada × 3. Identificar gargalos antes do usuário. Documentar baseline de performance para comparação futura.
perf
Infraestrutura
Containerização com Docker + multi-stage build
Dockerfile multi-stage (imagem de build ≠ imagem de produção). Non-root user. .dockerignore correto. docker-compose para ambiente de desenvolvimento local completo.
críticodevops
CI/CD Pipeline completo (GitHub Actions / GitLab CI)
On push: lint → test → build → security scan → deploy para staging. On tag: deploy para produção com aprovação manual. Rollback automatizado se health check falha.
críticodevops
IaC: Infraestrutura como Código
Terraform, Pulumi ou CDK. Nunca clicar no console AWS/GCP para criar infra. State remoto (S3 + DynamoDB). Plan em PR, Apply no merge. Infraestrutura revisada como código.
devops
Ambientes isolados: dev / staging / produção
Staging com dados anonimizados (nunca dados reais de prod). Variáveis de ambiente por ambiente. Feature branch deployments para review automático.
devops
Estratégia de deploy: Blue/Green ou Canary
Blue/Green: dois ambientes idênticos, switch de tráfego. Canary: 5% → 25% → 100% com monitoramento automático de error rate. Zero-downtime deployment.
reliability

Observabilidade & Incidentes
Operação e Manutenção
0/0
▼
Os Três Pilares da Observabilidade
Logging estruturado (JSON) com níveis e correlação
Winston, Pino, Loguru. Campos obrigatórios: timestamp, level, message, service, trace_id, request_id. Nunca log de dados sensíveis (PII). Centralizado: Datadog/Loki/CloudWatch.
críticoobserv
Métricas com Prometheus/OpenTelemetry
RED: Rate (requests/s), Errors (% errors), Duration (latência). USE: Utilization, Saturation, Errors para infra. Dashboards Grafana. Alertas no PagerDuty/OpsGenie.
críticoobserv
Distributed Tracing (OpenTelemetry + Jaeger/Tempo)
Trace ID propagado em todos os serviços via headers. Span por operação relevante. Visualização do caminho completo de uma requisição. Essencial em microsserviços.
observ
Gestão de Incidentes
Runbook por componente crítico
Playbook de "o que fazer se X falhar": passos de diagnóstico, comandos de rollback, quem escalar. Sem runbook = engenheiro sênior acordado às 3h reinventando a roda.
críticoops
Post-mortem blameless após incidentes P1/P2
Cultura de SRE: o sistema falhou, não a pessoa. Timeline do incidente, causa raiz, 5 Whys, ações corretivas com owner e prazo. Compartilhado com o time.
cultura
Alertas com severidade e SLO burn rate
Alertas em 3 níveis: info (Slack), warning (email), critical (PagerDuty). Error budget burn rate: alertar quando 2% do budget semanal é consumido em 1h.
observ

Documentação & Apresentação
Comunicação e Legado
0/0
▼
README Profissional
README que funciona como portfólio técnico
Badges (CI status, coverage, versão). Demo GIF ou screenshot. Problema que resolve + decisões de arquitetura chave. Setup em < 3 comandos. Links para ADRs e Design Doc.
críticodocs
CONTRIBUTING.md e CODE_OF_CONDUCT.md
Como contribuir: branch naming, commit convention, como rodar os testes. Demonstra maturidade de projeto open-source. Requisito mínimo para qualquer repo público.
docs
Documentação técnica além do README
/docs: arquitetura, fluxos de negócio, guias de operação. Docusaurus, MkDocs ou Notion. "Docs as code" — documentação versionada junto com o código.
docs
IA & Padrões Modernos
Prompts versionados e testados como código
Se o projeto usa LLMs: prompts em arquivos versionados, não hardcoded strings. Evals automatizados (LangSmith, PromptFoo). Fallback se o modelo falhar ou mudar.
ai
Observabilidade para pipelines de IA/ML
Rastrear: latência do LLM, custo por chamada, qualidade de output (thumbs up/down), hallucination rate. LangSmith, Helicone ou Braintrust para LLM observability.
ai
Acessibilidade (WCAG 2.1 AA) para UIs
Contraste mínimo 4.5:1. Navegação por teclado. ARIA labels. axe-core no CI. Não é só ético — é legal em vários países e afeta SEO. Stripe e Airbnb são referência.
a11y
FinOps: custo de infra monitorado desde o dia 1
Budget alerts na AWS/GCP. Tags de custo por serviço/ambiente. Rightsizing automatizado. Revisão de custo mensal. Projetos escalam em custo tão rápido quanto em features.
finops
Semantic Versioning (SemVer) aplicado
MAJOR.MINOR.PATCH. Breaking change → MAJOR. Nova feature → MINOR. Bug fix → PATCH. Release tags no Git. Package.json/pyproject.toml atualizado automaticamente pelo CI.
docs
